# JR 乗車 RECORD

JR 일본 철도 **승차 기록 PWA**. 지도에서 노선을 보고, 직접 탄 구간을 색칠해 기록하는 개인용 웹앱입니다.

- **라이브:** https://pos09009.github.io/jr-riding-record/
- 휴대폰에서 브라우저로 접속 → "홈 화면에 추가"로 앱처럼 설치 (오프라인 동작)

## 기능

- 캔버스 기반 커스텀 지도 (외부 지도 API 없이 동작, 팬/줌)
- 구간별 승차 기록 (지도 팝업 토글 · 수동 범위 입력 · GPS 자동 감지)
- 역/노선 이름 검색 → 해당 역으로 지도 확대
- **크레딧 모델**: 각역정차로 완주하면 그룹 내 쾌속·중거리도 자동 인정 (조반·츄오·소부 그룹). 쾌속이 건너뛴 구간은 미인정
- **직결/통합 보기**: 우에노도쿄라인·쇼난신주쿠라인처럼 여러 노선을 잇는 운행 계통은 별도 색 선을 긋지 않고 "직결" 버튼으로 관련 노선을 한 번에 보여줌. 츄오 본선·조반선·소부 본선·신에츠 본선처럼 실제로는 쾌속+완행+지선(+사철 이관 구간)으로 나뉜 노선도 "통합 보기"로 전체 구간을 확인 가능
- **환승역 자동 감지**: 같은 좌표에 두 노선 이상이 겹치면 지도에 환승 마커(흰 원)로 표시, 클릭 시 그 역을 지나는 모든 노선과 인접 구간을 팝업으로
- **비JR 구간 처리**: 사철 소유 구간(소테츠·JR직통선의 하자와요코하마코쿠다이~에비나 구간, 신에츠 본선의 시나노 철도·에치고토키메키 철도 구간 등)과 폐선 구간(신에츠 본선 요코카와～카루이자와, 현재 JR버스 대체 운행)은 지도엔 그려지거나(사철) 아예 빈틈으로 남고(폐선), 탑승 기록은 가능해도 완주율 집계에서는 제외 — 각각 안내 문구로 설명
- 백업: 기록 JSON 내보내기 / 가져오기 (구버전 백업도 자동 변환)
- PWA: 오프라인 캐시, 홈 화면 설치

## 파일 구조

| 파일 | 역할 |
|------|------|
| `index.html` | 화면·스타일·PWA 메타 (**진입점**) |
| `app.js` | 모든 로직 (지도·GPS·검색·기록·크레딧 계산) |
| `data/kanto.js` | 간토 노선 데이터 (원본 10개 노선) |
| `data/kanto2.js` | 간토 노선 데이터 (2차 이후 확장분, 지속 증가) |
| `data/services.js` | **운행 계통 데이터** — 크레딧 그룹, 통합 보기, 직결 매핑, 물리 공유 구간. `window.JR_SERVICES`로 노출, app.js보다 먼저 로드되어야 함 |
| `sw.js` | 서비스워커 (오프라인 캐시) |
| `manifest.json` | PWA 매니페스트 |
| `tools/validate.js` | 데이터 검증 러너 (로컬 전용, 배포 불필요 — 아래 참조) |

데이터는 `<script src>`로 불러와 `window.JR_DATA`에 합쳐집니다 (`file://`에서도 동작하도록 `fetch` 대신 이 방식 사용). 노선 = `{id, name, color, loop, stations:[{n, lat, lng}, ...]}`. 선택 필드:
- `code`/`codeColor`: 공식 노선기호 배지, `badges:[{code,color,filled}]`로 멀티 배지도 가능
- `drawSkip:[i,...]`: 이 구간은 다른 노선(직결 상대)이 대신 그림
- `tailFrom`/`tailColor`: 이 인덱스부터 끝까지 다른 색(예: 소테츠의 相鉄 구간)
- `nonJrFrom`: 이 인덱스부터 끝까지는 실제 JR 소유 선로가 아님, 노선 그려지고 탑승 기록도 되지만 **완주율 집계에서 제외**
- `showWith:[lid 또는 {id,from,to}]`: 이 노선 선택 시 같이 표시할 다른 노선(구간 지정 가능)
- `via:[{n,lat,lng}]`: 정차하지 않고 통과하는 경유 웨이포인트(지선 분기점 등), 역 인덱스·환승 판정에서 제외되고 폴리라인만 꺾어 그림
- `note`: 노선 선택 시 지도 좌상단에 뜨는 안내 문구

## 기록 키 (segmentId)

구간 완료 여부는 `done[key]`에 저장되며, 키는 **`노선id:역A원시좌표~역B원시좌표`**(좌표 오름차순 정준 정렬) 형식입니다. 순환선(`loop:true`)은 마지막 역→첫 역 구간이 추가됩니다.

이 키는 **역 인덱스가 아니라 역의 실제 좌표를 기반**으로 하기 때문에, 노선 중간에 역이 추가/삭제되거나 순서가 바뀌어도 기존 기록이 밀리지 않습니다(v111 이전에는 `노선id_인덱스` 방식이라 중간역 삽입 시 그 뒤 모든 기록이 어긋나는 문제가 있었음). 구버전 기록은 앱 로드 시 `migrateDone()`이 자동으로 새 형식으로 변환합니다.

## 역명 표기 원칙

일본어 **원음 우선**, 단 한국어에서 굳어진 관용 표기는 예외.

- 무성음은 격음으로: 高崎→타카사키(다카사키 ❌), 吉祥寺→키치조지
- 한자 음독 금지: 自治医大→지치이다이, 新都心→신토신
- ㅈ·ㅊ 뒤 요음의 y는 생략 (ㅈ·ㅊ이 이미 구개음): 町(ちょう)→초, じゅ→주 (쵸·쥬 ❌). 단 ㅅ은 유지(しゃ→샤)
- 장음 생략: 国府津(こうづ)→코즈 (코우즈 ❌), 下総(しもうさ)→시모사 (시모우사 ❌)
- おお(오단+お)도 こう(오단+う)와 동일하게 장음 축약: 大宮(おおみや)→**오미야**(오오미야 ❌, JR 공식 로마자도 매크론 하나 Ōmiya). お+お 표기는 大·氷·十 등 소수 고유어의 역사적 관습일 뿐 발음은 こう와 동일한 장음
- 단 같은 모음 반복(いい・しい・にい)은 장음 생략과 다른 케이스 — 둘 다 표기: 飯倉→이이구라, 椎柴→시이시바, 新治→니이하리 (나무위키의 이구라·시시바·니하리는 장음 생략한 비표준, JR 공식 로마자 Iigura/Shiishiba/Niihari와 대조해 확인)
- 역명 읽기는 한자만 보고 추측 금지 — 위키로 개별 확인 (같은 한자도 역마다 읽음이 다름)
- 공식 역명의 가운뎃점(・)은 한글 표기에도 유지: 万座・鹿沢口→만자·카자와구치
- 관용 예외: 도쿄(東京) · 도카이도/케이힌**도**호쿠(東) · 츄오(中央, 노선명만) · 니가타(新潟, にい 반복 규칙상 "니이가타"가 되어야 하나 도쿄급으로 굳어진 관용 채택 — 단 新津(にいつ)는 관용 아니라 규칙대로 니이츠)

## 라인컬러 검증 원칙

임의로 색을 넣지 않고 위키피디아 노선 문서 인포박스의 `路線色=#xxxxxx`를 1순위로 채택, 영어 위키 `color` 주석으로 교차 확인. **단 실제 공식 노선도 사진이 있을 경우 해당 색을 우선.** 노선 색상 관련 위키 인포박스 값이 실제 안내색과 다른 경우가 종종 있음(료모선·소토보선·우치보선·쿠루리선·미토선·아가츠마선 등에서 실제로 발견).

## 데이터 검증

노선/계통 데이터를 추가·수정한 뒤에는 아래를 실행해 참조 오류를 잡습니다 (Node 필요):

```
node tools/validate.js
```

검사 항목: 노선 id 중복, `LINE_META` 정합, 역 필드 무결성, 좌표 간격 이상치(>30km 또는 <100m), `drawSkip`/`tailFrom`/`nonJrFrom` 범위, `data/services.js`의 크레딧 그룹·통합 보기·직결 매핑·공유 구간이 참조하는 노선 id·역명이 실제로 존재하는지. 앱 자체도 로드 시 `validateData()`를 실행해 브라우저 콘솔에 같은 검사를 띄웁니다(`window.__dataIssues`).

## 새 노선 추가

1. 위키피디아에서 역 목록·좌표(coordinates API)·라인컬러(`路線色`)를 전수 취득
2. `data/kanto.js` 또는 `data/kanto2.js`에 `{id, name, color, loop, stations:[...]}` 추가, 기존 노선과 만나는 역은 좌표를 그대로 공유(자동 환승 감지)
3. 직결/통합/크레딧 그룹에 얽히면 `data/services.js`에 항목 추가(노선 데이터 자체는 건드리지 않음)
4. `node tools/validate.js`로 검증
5. `sw.js`의 `CACHE` 버전 숫자 올리기

## 새 지역 파일 추가

1. `data/지역명.js` 생성 → `window.JR_DATA.지역키 = [ {id, name, color, loop, stations:[...]}, ... ]`
2. `index.html`에 `<script src="data/지역명.js"></script>` 한 줄 추가 (`services.js`보다 앞)
3. `sw.js`의 `ASSETS`에 파일 경로 추가 + `CACHE` 버전 숫자 올리기

## 업데이트 반영 (PWA 캐시 주의)

데이터/코드 수정 후 GitHub에 올렸는데 폰에서 안 바뀌면 캐시 문제입니다 (GitHub Pages 약 10분 HTTP 캐시 + 옛 서비스워커). 가장 확실한 방법: **홈 화면에서 앱 삭제 → 브라우저로 재접속 → 다시 홈 화면에 추가**. 파일 수정 시 `sw.js`의 `CACHE` 버전을 항상 올리세요. `data/services.js`를 새로 추가한 경우, 업로드 시 빠뜨리면 크레딧/통합/직결 기능이 전부 비활성화되니 주의.
